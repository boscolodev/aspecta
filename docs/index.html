aspecta

Biblioteca Spring Boot para tratamento centralizado de exceções e logging AOP automático com suporte a i18n, mascaramento de dados sensíveis e processamento assíncrono.

Spring Boot 3.1+ Java 21 AOP / AspectJ io.github.boscolodev Async Logging i18n Sensitive Data Masking
🎯

Visão Geral

Aspecta elimina boilerplate nos seus projetos Spring Boot, entregando dois pilares:

🛡️

Tratamento de Exceções

Handler global @RestControllerAdvice com respostas padronizadas (BASIC e COMPLETE), fábrica de exceções e suporte a Bean Validation.

📝

Logging AOP Automático

Anote qualquer método com @LogOn e obtenha entrada, saída e erros registrados automaticamente, com mascaramento de dados sensíveis.

🌍

Internacionalização

Mensagens de log em Português (pt-BR) e Inglês (en) com fallback automático. Configurável via messages.properties.

⚡

Assíncrono por Padrão

Logs executados em thread separada via Spring @Async, garantindo que o fluxo principal não seja bloqueado.

📦

Instalação

Adicione a dependência ao seu pom.xml:

Maven · pom.xml 
<dependency>
  <groupId>io.github.boscolodev</groupId>
  <artifactId>aspecta</artifactId>
  <version>1.0.0</version>
</dependency>
Nota: A publicação no Maven Central está em andamento. Enquanto isso, instale localmente com mvn install.

Configuração

Todas as propriedades são opcionais. O comportamento padrão funciona sem nenhuma configuração.

YAML · application.yml 
logger:
  enabled: true               # Liga/desliga todo o logging
  enable-i18n: true          # Ativa mensagens internacionalizadas
  projectName: MEU-PROJETO   # Aparece no prefixo dos logs
  structured-output: false      # true → emite JSON indexável (Loki/Datadog)
  sensitive-keys:             # Campos mascarados globalmente (padrão já inclui os abaixo)
    - password
    - cpf
    - token
    - secret
    - apikey
    - accesstoken
    - refreshtoken
    - authorization
    - privatekey
    - creditcard
  expose-exception-details: false  # true apenas em dev — nunca em produção
  async:                       # Configuração do executor dedicado
    core-pool-size: 2           # min: 1, max: 20
    max-pool-size: 8            # min: 1, max: 100
    queue-capacity: 500         # min: 1, max: 5000
Propriedade Tipo Padrão Descrição
logger.enabled Boolean true Habilita ou desabilita o logging AOP globalmente
logger.enable-i18n Boolean true Usa mensagens do bundle i18n ou o provider padrão
logger.projectName String ASPECTA Nome exibido como prefixo em todos os logs
logger.sensitive-keys List<String> password, cpf, token, secret, apikey, accesstoken, refreshtoken, authorization, privatekey, creditcard Chaves JSON cujos valores serão substituídos por *** (strings e números). Comparação case-insensitive.
logger.structured-output Boolean false Quando true, cada log é emitido como JSON com campos event, project, class, method, args/result/exception, durationMs e traceId (se presente no MDC).
logger.expose-exception-details Boolean false Quando true, o campo details do CompleteResponse inclui a mensagem do cause. Habilitar apenas em ambientes não-produção.
logger.async.core-pool-size Integer 2 Threads base do executor aspectaLoggerExecutor. Limites: 1–20
logger.async.max-pool-size Integer 8 Máximo de threads simultâneos no pool de logging. Limites: 1–100
logger.async.queue-capacity Integer 500 Capacidade da fila interna. Ao estourar, aplica CallerRunsPolicy. Limites: 1–5000

Tipos de Resposta — ExceptionType

O enum ExceptionType define o formato da resposta retornada ao cliente quando uma exceção é lançada.

BASIC

Resposta mínima com status e mensagem. Ideal para erros simples onde não há necessidade de expor detalhes internos.

JSON
{
  "status": "404",
  "message": "Usuário não encontrado"
}

COMPLETE

Resposta completa com detalhes, path e timestamp. Recomendado para ambientes de desenvolvimento e erros que precisam de rastreabilidade.

JSON
{
  "status": "500",
  "message": "Erro interno",
  "details": "NullPointerException",
  "path": "/api/user",
  "timestamp": "2025-05-30T12:34:56"
}
🏭

ApiExceptionFactory

Classe utilitária com método estático troll() para lançar exceções padronizadas de forma fluente.

Java 
// Lança com status 500 (padrão) e causa raiz
throw ApiExceptionFactory.troll(
    ExceptionType.BASIC,
    "Falha na operação.",
    exception
);

// Lança com HTTP status personalizado
throw ApiExceptionFactory.troll(
    ExceptionType.BASIC,
    "Entrada inválida.",
    HttpStatus.BAD_REQUEST
);

// Lança com status, causa e resposta completa
throw ApiExceptionFactory.troll(
    ExceptionType.COMPLETE,
    "Serviço indisponível.",
    HttpStatus.SERVICE_UNAVAILABLE,
    exception
);
SobrecargaStatus padrãoInclui causa
troll(type, message, cause) 500 Internal Error Sim
troll(type, message, status) Personalizado Não
troll(type, message, status, cause) Personalizado Sim

Response DTOs

BasicResponse

Campos: status, message

Usado por ExceptionType.BASIC. Resposta mínima sem dados de diagnóstico.

CompleteResponse

Campos: status, message, details, path, timestamp

Usado por ExceptionType.COMPLETE. Inclui rastreabilidade completa.

FieldMessage

Campos: fieldName, message, details

Estende BasicResponse. Cada item representa um campo inválido na validação.

Anotação @LogOn

Marque qualquer método Spring com @LogOn para ativar o logging automático via AOP.

Java 
@Service
public class UserService {

    // Mascara campos sensíveis (padrão)
    @LogOn(sensitiveData = true)
    public User createUser(String name, String email, String password) {
        return userRepository.save(new User(name, email, password));
    }

    // Sem mascaramento — dados públicos
    @LogOn(sensitiveData = false)
    public User findUser(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
}

@RestController
@RequestMapping("/api/user")
public class UserController {

    @GetMapping("/{id}")
    @LogOn // sensitiveData = true por padrão
    public ResponseEntity<User> findUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.findUser(id));
    }
}
ParâmetroTipoPadrãoDescrição
sensitiveData boolean true Se true, aplica mascaramento nas chaves configuradas em logger.sensitive-keys
level LogLevel INFO Nível de log (DEBUG, INFO, WARN). Se o nível estiver desativado no framework de logging, nenhuma serialização ocorre
logStackTrace boolean false Se true, loga o stack trace completo quando o método lança uma exceção
excludeFromMask String[] {} Campos que não serão mascarados neste método, mesmo que estejam em logger.sensitive-keys. Útil para exibir dados específicos em métodos de diagnóstico

LoggerAspect — Fluxo Interno

O LoggerAspect usa @Around para envolver qualquer método anotado com @LogOn.

@LogOn detectado
→

→
Sanitiza dados
→
logInfo (entry)
→

→
logInfo (exit) ou logError
Saída de Log — Exemplo 
# Entrada
[MEU-PROJETO][UserService] Entering method: createUser() with | Args: [name=John, password=***]

# Saída
[MEU-PROJETO][UserService] Exiting method: createUser() returned | Return: User{id=1, name=John}

# Erro
[MEU-PROJETO][UserService] Error in method createUser: IllegalStateException - Simulated error.

O template padrão é [{projectName}][{className}] {message}.

Mascaramento de Dados Sensíveis

O SensitiveDataMasker usa regex case-insensitive para substituir valores de campos sensíveis por ***, cobrindo tanto strings quanto valores numéricos JSON. O mascaramento é aplicado nos argumentos de entrada e no valor de retorno.

Strings e números — ambos mascarados 
// Input: { "name": "João", "password": "abc123", "cpf": 12345678900 }
// Log:   { "name": "João", "password": "***",   "cpf": "***" }
Padrões regex (por chave):
Strings: (?i)("key"\s*:\s*")[^"]*"
Números: (?i)("key"\s*:\s*)(-?\d+(?:\.\d+)?)
As chaves são escapadas com Pattern.quote() para prevenir injeção de regex / ReDoS.

excludeFromMask — exibir campos específicos

Use excludeFromMask no @LogOn para exibir determinados campos mesmo que estejam em logger.sensitive-keys. Útil para métodos de diagnóstico ou auditoria controlada.

Java — exemplo 
// Classe: @LogOn(sensitiveData = true) — mascara tudo
@Service
@LogOn(sensitiveData = true)
public class AuditService {

    // Método de diagnóstico: mostra email, mascara password/cpf/token
    @LogOn(sensitiveData = true, excludeFromMask = {"email"})
    public void auditLogin(String email, String password) { ... }

    // Método normal: mascara tudo (herda da classe)
    public void createUser(String email, String password, String cpf) { ... }
}

// Log de auditLogin: email=joao@ex.com, password=***
// Log de createUser: email=***, password=***, cpf=***

Chaves sensíveis padrão: password, cpf, token, secret, apikey, accesstoken, refreshtoken, authorization, privatekey, creditcard. Extensível via logger.sensitive-keys. O mascaramento cobre JSON ("key":"value"), texto plano (key=value, key: value) e padrões customizados via MaskingStrategy.

⚡

Logging Assíncrono

Todo o logging é delegado ao AsyncLoggerService, que usa @Async("aspectaLoggerExecutor") em um executor dedicado e nomeado, sem interferir no executor @Async da aplicação consumidora.

AsyncLogger interface

Quatro métodos: logDebug, logInfo, logWarn, logError — cada um delegado ao nível SLF4J correspondente.

aspectaLoggerExecutor

Bean ThreadPoolTaskExecutor criado por AsyncLoggerConfig com prefixo de thread aspecta-logger-*. Configurável via logger.async.*.

CallerRunsPolicy

Ao estourar a fila (queueCapacity), a tarefa de log é executada na thread chamadora em vez de ser descartada — sem perda de logs sob pressão.

Guard de serialização

Se o nível de log configurado em @LogOn(level=...) estiver desativado no framework (ex: DEBUG em produção), nenhuma serialização de args ocorre — zero overhead.

Propagação de contexto MDC

O AsyncLoggerConfig usa um TaskDecorator que copia automaticamente o MDC da thread chamadora para a thread do pool — traceId e outros campos de contexto são preservados fim-a-fim, mesmo em logs assíncronos.

Isolamento garantido: O Aspecta usa @Async("aspectaLoggerExecutor") — nunca o executor padrão do Spring. A configuração @Async da sua aplicação permanece intacta.

Internacionalização (i18n)

O Aspecta suporta mensagens de log em múltiplos idiomas via ResourceBundleMessageSource.

messages.properties

Arquivo padrão (Português). Fallback quando nenhum locale específico é encontrado.

messages_en.properties

Mensagens em Inglês para locales en, en-US, etc.

messages_pt_BR.properties

Mensagens em Português Brasileiro (locale padrão configurado).

messages_en.properties 
log.entry=Entering method: {0}() with | Args: {1}
log.exit=Exiting method: {0}() returned | Return: {1}
log.error=Error in method {0}: {1} - {2}

O DelegatingMessageProvider usa o padrão Strategy para alternar entre I18nMessageProvider e DefaultMessageProvider conforme logger.enable-i18n.

Altas

Issues que degradam robustez, cobertura de mascaramento ou comportamento esperado sob condições adversas.

🔴

Críticas

  • Corrigido

    AsyncLoggerConfig sobrescrevia o executor global da aplicação

    A implementação de AsyncConfigurer em AsyncLoggerConfig substituía o executor padrão de todos os @Async da aplicação consumidora pelo pool da biblioteca, causando potencial starvation de threads, quebra de configurações de retry/timeout e comportamento imprevisível.

    Antes (vulnerável)
    public class AsyncLoggerConfig implements AsyncConfigurer {
    @Override public Executor getAsyncExecutor() { ... } // global!
}
    Depois (corrigido)
    @Bean(name = "aspectaLoggerExecutor")
public TaskExecutor aspectaLoggerExecutor() { ... }
// AsyncLoggerService usa @Async("aspectaLoggerExecutor") — isolado

    Impacto: Corrupção silenciosa do comportamento assíncrono de toda a aplicação. Corrigido com bean nomeado e @Async("aspectaLoggerExecutor").

  • Corrigido

    Injeção de Regex via sensitive-keys — ReDoS

    As chaves configuradas em logger.sensitive-keys eram interpoladas diretamente no padrão regex sem escapamento. Um valor como "pass.ord" tornava-se um regex funcional e poderia mascarar campos não intencionais; valores como "(a+)+" abriam vetor de ReDoS (Regular Expression Denial of Service), podendo travar a JVM em strings de entrada cuidadosamente construídas.

    Antes (vulnerável)
    // key inserido cru no padrão — qualquer metacaracter é executado
json.replaceAll("(?i)(\"" + key + "\"\\s*:\\s*\")[^\"]+", "$1***\"");
    Depois (corrigido)
    String quotedKey = Pattern.quote(key); // escapa todos os metacaracteres
json.replaceAll("(?i)(\"" + quotedKey + "\"\\s*:\\s*\")[^\"]*\"", "$1***\"");

    Impacto: Disponibilidade (DoS) e bypass de mascaramento. Fix: Pattern.quote(key) em SensitiveDataMasker.

🟠

Altas

  • Corrigido

    Valores numéricos JSON não mascarados

    O regex original (?i)("key"\s*:\s*")[^"]+" exige que o valor esteja entre aspas duplas. Valores numéricos como "cpf": 12345678900 ou "token": 9999 (sem aspas) não eram mascarados, expondo dados sensíveis nos logs mesmo com sensitiveData = true.

    Cenário — antes
    // Input: { "cpf": 12345678900, "name": "João" }
// Log (errado): { "cpf": 12345678900, "name": "João" }  ← CPF exposto!
    Fix aplicado
    // Segunda passagem para valores numéricos
json.replaceAll("(?i)(\"" + quotedKey + "\"\\s*:\\s*)(-?\\d+(?:\\.\\d+)?)", "$1\"***\"");
  • Corrigido

    Valores de retorno não são mascarados

    Apenas os argumentos de entrada passavam pelo SensitiveDataMasker. O valor de retorno era serializado via String.valueOf(result) sem filtro, expondo campos como password ou cpf no log de saída.

    Corrigido: masker.sanitizeResult(result, sensitiveData, excludeFromMask) é aplicado ao retorno antes de passar ao exitMessage.

  • Corrigido

    CompleteResponse.details controlado por logger.expose-exception-details

    O campo details de CompleteResponse era preenchido com cause.getMessage() da exceção raiz sem nenhuma sanitização, podendo conter queries SQL, file paths ou credenciais de conexão.

    Corrigido: A propriedade logger.expose-exception-details (padrão false) controla se o cause é exposto. Em produção o campo details é sempre null. Habilite apenas em ambientes não-produção:

    YAML — desenvolvimento
    logger:
  expose-exception-details: true  # nunca habilitar em produção
  • Corrigido

    Mensagens de exceção logadas sem mascaramento

    O LoggerAspect logava o erro com ex.getMessage() diretamente, sem passar pelo SensitiveDataMasker. Dados sensíveis em mensagens de exceção (ex: campos de constraint JPA) apareciam em plain text.

    Corrigido: O bloco de catch em LoggerAspect.doLog() agora aplica masker.sanitizeMessage(ex.getMessage(), sensitiveData, excludeFromMask) antes de passar a mensagem ao provider. Testado em LoggerAspectIntegrationTest.exceptionMessage_sensitiveData_isMaskedInErrorLog().

🟡

Médias

  • Corrigido

    Recursão sem limite em extractStatus

    O método extractStatus percorria a cadeia de causas via recursão sem limite de profundidade, abrindo vetor de StackOverflowError com exceções encadeadas profundamente ou circulares.

    Corrigido: Convertido para loop iterativo com limite de 10 níveis (MAX_CAUSE_DEPTH = 10) em GlobalExceptionHandler.

  • Corrigido

    Mensagem de validação hardcoded em Português

    A constante METHOD_ARGUMENT_MESSAGE estava hardcoded em Português, quebrando a consistência de i18n para APIs consumidas por clientes internacionais.

    Corrigido: Mensagem movida para os bundles (exception.validation.error) e resolvida via MessageSource com request.getLocale().

  • Corrigido

    @LogOn no próprio GlobalExceptionHandler

    O @LogOn foi removido do GlobalExceptionHandler. Handlers de exceção não devem ser interceptados pelo aspecto de logging — evitando serialização custosa de MethodArgumentNotValidException e chamadas ao executor assíncrono durante shutdown.

  • Corrigido

    Escalonamento de nível de log em logStackTrace

    Quando logStackTrace = true e o nível padrão level = INFO estava em uso, o stack trace era emitido em nível ERROR em vez de INFO, disparando alertas não intencionais.

    Corrigido: Adicionado case INFO -> log.info("Stack trace:", ex) explícito no switch de logStackTrace(). O método é package-private para permitir testes diretos em LoggerAspectStackTraceTest.

  • Corrigido

    Mascaramento falha em arrays e objetos não-JSON

    O SensitiveDataMasker usava String.valueOf(arg), produzindo [Ljava.lang.String;@hash] para arrays — sem campos JSON para mascarar.

    Corrigido: O método toStringArg(Object) agora despacha por tipo: Object[] → Arrays.deepToString(), arrays primitivos → Arrays.toString(), Collection e Map → serialização direta. Coberto por SensitiveDataMaskerTest.SanitizeArgsCollections.

  • Corrigido

    Lista padrão de sensitive-keys incompleta

    O default anterior cobria apenas password, senha, cpf, cnpj, token.

Riscos de Segurança das Melhorias Propostas

As funcionalidades implementadas introduzem vetores de atenção que devem ser considerados ao configurar e operar a biblioteca em produção.

  • Sem ação necessária

    Correlation ID / Trace ID — responsabilidade operacional, não da lib

    O traceId do MDC é incluído apenas nos logs internos — nunca em campos de resposta HTTP. Nenhum DTO (BasicResponse, CompleteResponse, ProblemDetail) expõe trace ou span IDs. O risco é exclusivamente operacional: se o sistema de agregação de logs (Kibana, Loki, etc.) estiver acessível sem autenticação, os trace IDs nos logs ficam visíveis. Isso é responsabilidade de quem opera a infraestrutura, não da lib.

  • Sem ação necessária

    Endpoint Actuator /actuator/aspecta — proteção via Spring Security

    O endpoint não expõe a lista de sensitive-keys — a lista é intencionalmente omitida. O endpoint expõe apenas estado operacional: logging habilitado, nome do projeto, i18n, structured-output e métricas do pool assíncrono (activeThreads, queueSize, queueUsagePct). A proteção do actuator com autenticação e autorização é responsabilidade do projeto consumidor via Spring Security, seguindo a mesma prática recomendada para qualquer endpoint /actuator/*.

  • Corrigido

    Thread pool do logger assíncrono — limites de configuração validados

    Valores mal configurados em logger.async.* (ex: queue-capacity: 0, max-pool-size: 1000) podiam causar exaustão de memória.

    Corrigido: LoggerProperties é anotado com @Validated e a inner class Async usa constraints JSR-380. A aplicação falha no startup com mensagem clara se algum valor estiver fora dos limites: core-pool-size (1–20), max-pool-size (1–100), queue-capacity (1–5000).

  • Corrigido

    Handlers de exceções do Spring não expõem mensagens internas

    Os handlers de HttpMessageNotReadableException e DataIntegrityViolationException não declaram a exceção como parâmetro — por design. Isso impede que qualquer mensagem interna (nomes de tabelas, colunas, detalhes de parsing) seja acidentalmente incluída na resposta. As respostas retornam apenas mensagens i18n genéricas via MessageSource.

    Testes de contrato em ExceptionHandlerSanitizationTest verificam que os bodies de resposta não contêm termos como constraint, duplicate key, JsonParseException, etc.

public interface MaskingStrategy {
    String apply(String input);
}

// Uso — registrar como @Bean no projeto consumidor:
@Bean

}
